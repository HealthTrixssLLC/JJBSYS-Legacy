EXCI Bridge in Plain English

A high-level explanation of how batch jobs can call CICS through a bridge model instead of operating on CICS resources directly.

EXCI is a bridge concept that lets a batch or non-CICS process request work from CICS in a controlled and structured way. Instead of manipulating CICS resources directly, the external caller asks a CICS-side program or transaction to perform the work and return a response.

Conceptual Flow

Batch Job
  ->
EXCI Client
  ->
CICS Transaction / Server Program
  ->
CICS File / Queue / Program
  ->
Response

Why use a bridge

It centralizes CICS-side handling and keeps external callers from depending on direct access to internal resource details.

What reviewers should look for

Request structures, COMMAREAs, routing behavior, transaction naming, queue use, and error-handling conventions.

Why it matters today

It is a useful historical pattern for understanding how legacy enterprises exposed controlled transaction access before modern API platforms.
